# CodeMentor AI：简历与面试材料

## 简历项目描述

**CodeMentor AI - 基于 RAG 与证据型长期记忆的 Java 个性化学习助手**  
`Dify Cloud Chatflow | FastAPI | Supabase | Render | LLM | RAG`

- 构建 Java AI Tutor，整合教材知识问答、课程规划、面试练习、继续学习和学习报告；将教材正文、课程地图和原子技能图谱拆分为独立知识库，降低目录检索对知识问答的干扰。
- 设计证据驱动的长期学习记忆：仅将练习题和面试题的结构化评估写入 Supabase，后端按“至少 3 次评估且正确率 >= 60%”确定主题掌握状态，避免模型或用户自述直接篡改学习进度。
- 使用 FastAPI 对接 Dify 与 Supabase，实现待评估题、评测证据、主题/技能状态和学习报告 API；部署于 Render，并通过 `X-Memory-Token` 隔离服务端密钥。
- 实现跨会话继续学习与学习报告：根据 `learning`、`reviewing`、`mastered` 状态、原子技能正确率和课程地图顺序，生成定向复习与下一步建议。

## 一分钟项目介绍

这是一个面向 Java 初学者的 AI 学习助手。它不是普通的问答机器人：教材问答使用 RAG，学习顺序由显式课程地图控制；练习和面试答案会被结构化评估，后端把证据写入长期 Memory。只有满足明确阈值的答题记录才会把主题标为已掌握。用户下次打开应用时，可以继续上次的学习、获得针对薄弱技能的练习，并查看由真实评估数据生成的学习报告。

## 高频面试问题

### 为什么不直接使用 Dify 的对话历史做 Memory？

对话历史同时包含闲聊、错误回答和重复内容，难以稳定检索，也不适合直接作为掌握状态依据。该项目保存的是主题状态、技能正确率、误区和最近学习内容等结构化摘要，体积更小、可验证且便于确定性计算。

### 为什么掌握状态不让 LLM 直接判断？

LLM 对“用户已经掌握”的判断不可重复，也容易受用户自述影响。项目将 LLM 的职责限定为评估单题并输出结构化证据，状态转换由 FastAPI 的固定阈值规则完成，便于测试和审计。

### 如何解决教材目录总被召回的问题？

将教材正文与课程地图分成两个知识库。知识问答与讲评检索知识点正文；学习计划和继续学习检索课程地图。课程地图使用稳定 topic_id 与 next_topic_id，而不是依赖模型从目录推断顺序。

### 为什么还需要原子技能图谱？

章节粒度过粗。例如 `if条件语句` 下，布尔表达式和 `if...else` 分支完整性可能掌握程度不同。原子技能使系统能够保存更细的正确率和误区，并在复习时定位具体薄弱点。

### 用户有待评估题时又请求学习报告，如何避免误记为错误？

在答题评估器前增加输入意图判断，区分 `answer`、`cancel` 与 `new_request`。对于报告、继续学习等新请求，先取消待评估题且不写分数，再进入常规意图路由。

### 学习报告如何避免统计幻觉？

课程总数、主题状态、正确率和完成率均由后端报告 API 确定性计算。报告 LLM 只能格式化 API 返回的数据，不能自行计算或声明不存在的学习成果。

## 可展示的测试场景

1. 请求一道 Java 条件判断练习题，系统创建待评估题。
2. 提交代码片段，系统识别为作答而不是未作答，并给出结构化讲评。
3. 再次请求学习报告，报告显示评估次数、正确率、技能误区和 `reviewing` 状态变化。
4. 在新会话输入“继续昨天学习”，系统从长期 Memory 读取上次主题并生成针对性复习安排。
